### 기능 5: 예산제약 Fractional Knapsack 배분 ⭐ 최우선

#### 핵심 개념

기존 SaaS(Netstock·EazyStock 등)는 **순방향**: 목표 서비스 수준 → 필요 재고량 산출  
본 시스템은 **역방향**: 고정 예산 → 가장 많은 가치를 지키는 SKU 조합 산출

이것이 IBM 특허 US6078900(1998, 만료)의 선행기술 위에 있는 구조이므로,  
알고리즘 자체보다 **해상 리스크 + ABC-XYZ + AI 해석 결합 SME 경량화**가 차별점임.

#### 입력값

| 변수 | 설명 |
|---|---|
| `budget` | 사용자 입력 — 총 예산 (원) |
| `sku_list` | SKU별 리스트 |
| `safety_stock_qty[i]` | 기능 2에서 계산된 SKU별 안전재고 수량 |
| `unit_cost[i]` | SKU별 단가 (원) |
| `value[i]` | SKU별 보호 가치 — ABC-XYZ 임계 가중치 × 매출 기여 |

#### 가치(V) 계산 기준

ABC-XYZ 셀 기반 임계 가중치 적용:

| 셀 | 가중치 | 이유 |
|---|---|---|
| AZ | 2.8x | 고매출·고변동 — 위기 시 손실 최대 |
| AX | 2.0x | 고매출·안정 |
| BY | 1.5x | 중간 |
| CX | 1.0x | 저매출·안정 — 기준값 |
| CZ | 0.8x | 저매출·고변동 — 보호 효율 낮음 |

`value[i] = 가중치[i] × 월평균매출[i]`

#### Fractional Knapsack 로직

```python
# SKU별 단위 가치 계산 (가치 / 비용)
for sku in sku_list:
    sku.cost = sku.safety_stock_qty * sku.unit_cost
    sku.ratio = sku.value / sku.cost  # 높을수록 우선

# 비율 내림차순 정렬 후 예산 소진
sorted_skus = sorted(sku_list, key=lambda x: x.ratio, reverse=True)

remaining_budget = budget
allocation = {}

for sku in sorted_skus:
    if remaining_budget <= 0:
        break
    if sku.cost <= remaining_budget:
        allocation[sku.id] = 1.0          # 전량 보호
        remaining_budget -= sku.cost
    else:
        fraction = remaining_budget / sku.cost
        allocation[sku.id] = fraction     # 부분 보호
        remaining_budget = 0
```

#### 출력값

- 보호 SKU 목록 + 보호 비율 (100% / 부분 / 0%)
- **포기 SKU 목록 + 예상 손실액** (기능 6과 연동)
- 예산 소진 현황 (총 예산 / 사용액 / 잔액)

#### 주의 사항 (이중계산 문제 — 확인 중)

현재 코드에서 위기배수 적용 위치 확인 필요:
- `Z × σL × 위기배수` → **이중계산 오류** (σL 자체가 이미 리드타임 변동성 포함)
- `μL × 위기배수` 후 안전재고 재계산 → **올바른 방식**

조원(희도) 확인 후 반영 예정.

### 기능 6: 포기 SKU + 예상 손실액 명시

#### 개념

예산 초과로 보호받지 못한 SKU에 대해 **위기 발생 시 예상 손실액**을 명시.  
의사결정자(SME 대표)가 "이 SKU를 포기하면 얼마를 잃는가"를 숫자로 확인.

#### 계산식

```python
# 포기 SKU의 예상 손실액
expected_loss[i] = (
    (1 - allocation[i])          # 미보호 비율
    * stockout_prob[i]           # 위기 시 재고 소진 확률 (시나리오별)
    * monthly_revenue[i]         # 월 매출
    * crisis_duration_months     # 위기 지속 기간 (기본값: 2개월)
)
```

#### 출력

- 포기 SKU 테이블: SKU명 / 미보호 비율 / 예상 손실액(원)
- 전체 포기 예상 손실 합계

### 기능 7: 항로 집중도 경고

#### 배경 및 근거

표준 σL 안전재고 공식은 SKU 간 공급차질 리스크를 **독립으로 가정**함  
(ScienceDirect 2024: *"It is common to assume supply disruption risks are independent. However, risks may be correlated."*)

동일 항로를 공유하는 SKU들은 위기 시 **동시 영향**을 받음 — 독립 가정 위반.  
IMF WP 2025(Ahn & Tan): 포트폴리오 이론 적용 — 지리적 집중도(HHI)↑ = 공급 충격 리스크↑ 실증.

#### 입력값

- SKU별 `route` 태그 (예: "상하이-부산", "상하이-LA", "호치민-부산")
- CSV 업로드 시 컬럼으로 받거나, UI에서 수동 입력

#### 계산: 항로 HHI

```python
from collections import Counter

route_budget = {}  # 항로별 배분 예산 합계
for sku in protected_skus:
    route_budget[sku.route] = route_budget.get(sku.route, 0) + allocation_cost[sku.id]

total = sum(route_budget.values())
shares = [v / total for v in route_budget.values()]
hhi = sum(s**2 for s in shares)  # 0~1, 높을수록 집중

# 경고 기준
# HHI > 0.5 → 위험 (단일 항로 50% 이상)
# HHI > 0.25 → 주의
```

#### 출력

- 항로별 예산 비중 파이차트 or 바차트
- HHI 수치 + 경고 메시지
  - 0.5 초과: 🔴 "단일 항로 집중 위험 — 홍해·수에즈 수준 위기 시 보호 예산의 XX%가 동시 무력화될 수 있습니다."
  - 0.25~0.5: 🟡 "항로 집중도 주의"
  - 0.25 이하: 🟢 "양호"

#### 중요 표현 제약

- "기존 도구들은 이 기능이 없습니다" → **사용 금지** (증명 불가)
- "표준 공식의 독립 가정 한계를 보완하는 경고 지표입니다" → **사용 가능**
