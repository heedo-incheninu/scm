# 11. 개발자 인수인계서

> **이 문서의 목적**: 조원분이 이 파일 하나만 읽어도 "무엇을 왜 만들어야 하는지"를 이해할 수 있도록 작성했습니다. 채팅 전체나 05~10번 문서를 다 안 읽으셔도 됩니다.
>
> **업데이트 방식**: 이 문서는 계속 갱신됩니다. 단, 조원분이 한 기능을 완료했다고 알려주실 때마다 다음 단계 내용으로 갱신하는 방식입니다(작업 중간에 임의로 바뀌지 않습니다).
>
> **프롬프트 관련**: 아래 기능별로 "AI에게 줄 참고용 프롬프트"가 있습니다. 이건 정답이 아니라 **참고용 예시**입니다. 어떤 AI도구를 쓰시든 본인 방식대로 수정해서 쓰시면 됩니다.

## 0. 한 줄 요약

> **해상 리드타임 위기 시, 제한된 예산 안에서 SKU별 안전재고 우선순위를 산출하는 한국 수입 SME용 의사결정 보조 MVP**

지금 GitHub에 있는 4모듈(ABC-XYZ 분류, σL 안전재고, 위기시나리오 배수, 예산제약 knapsack)은 **그대로 유지**합니다. 이번 인수인계는 그 위에 **3가지를 추가/수정**하는 작업입니다.

## 1. 왜 이렇게 됐는지 — 의사결정 히스토리 (요약)

조원분이 처음부터 다시 읽으실 필요 없게, 핵심 전환점만 정리합니다.

| 단계 | 무슨 일이 있었나 | 왜 중요한가 |
|---|---|---|
| 1 | 기존 4모듈 개발 완료(이전 작업) | 출발점. 이건 안 바뀝니다 |
| 2 | 11개 평가기준(고객/중요성/근본원인/공감대/시장크기 + 가정지양/증명가능성/사업현실성/차별성/아이디어/참신성)으로 주제를 재검증 | 처음엔 평균 4.9/10으로 약했음 |
| 3 | 5Why(왜 이 문제가 안 풀렸는가)를 캐다가, "한국 SME가 왜 기존 솔루션을 안 쓰는지"는 **인터뷰 없이는 모른다**는 게 드러남 | 이건 못 푸는 한계로 인정하고, "안다고 주장 안 함"으로 정리 |
| 4 | 기존 솔루션(Netstock/Lokad/Flowlity/EazyStock) 조사 결과, ABC-XYZ·동적안전재고를 **이미 다 하고 있음**을 확인 | "우리가 처음 만든 기능"이라는 주장은 위험. "기존 기법 조합+한국형 경량화"로 포지셔닝 전환 |
| 5 | 이 4개 업체의 한국어 매체 노출을 조사 → 거의 전무(1건만 단순 비교언급) | "시장공백"이라고 단정은 못하지만, "한국 접근성 검증이 안 됐다"는 근거로 사용 |
| 6 | 차별화 아이디어로 "GSCPI(공급망압력지수) 실시간 연동"을 검토했으나, 이게 진짜 독창적인지 평가한 결과 **"지표 모니터링 자체는 차별점이 아니다"**로 결론 | GSCPI는 핵심 차별점에서 **보조 검증기능으로 격하**됨(아래 기능2) |
| 7 | 대신 강의교재(SCM론 확률과 재고 챕터)의 **서비스수준(service level) 이론**을 활용한 아이디어로 전환 | 이게 지금 **최우선 핵심기능**(아래 기능1) |
| 8 | 타겟고객 숫자(처음엔 "222,925개"로 알고 있었음)가 **AI 조사과정의 환각**이었음을 발견, 통계청 원문을 직접 열어 정확한 숫자(213,494개 중 도소매업 125,122개)로 교체 | 보고서·발표에서 숫자 인용 시 반드시 10번 문서의 최신 수치 사용 |
| 9 | "왜 한국 SME가 대기업보다 더 불리한가"를 보강하려고 조사하다가, 해양수산부가 실제로 "중소기업 전용 선적공간" 정책을 운영 중이라는 **정부 1차자료**를 5건 확보 | 근본원인 보강 완료 |
| 10 | 위기배수(1.8배/2.2배) 산출 근거였던 구체수치(108일/110일)를 재검증하던 중, **확인이 잘 안 되는 상태로 남음**(현재진행중) | 아래 "주의사항" 참고 — 수치가 바뀔 수 있음 |

**결론**: 이 프로젝트는 "기존에 없는 걸 새로 만든다"가 아니라, **"검증된 위험데이터 + 검증된 알고리즘 이론을 한국 SME가 쓸 수 있는 경량 도구로 조합한다"**는 포지셔닝입니다. 개발하실 때도 이 톤을 기억해주세요 — 너무 화려한 기능보다, "왜 이 숫자가 나왔는지 설명 가능한" 쪽이 우선입니다.

## 2. 지금 만들어야 할 기능 3개

### 기능 1 (★최우선, 핵심 차별점) — 서비스수준 차등 선택형 다중선택 knapsack

**왜 필요한가**: 지금 knapsack은 SKU를 "보호함/포기함" 이진택일로만 고릅니다. 이걸 "이 SKU는 90% 서비스수준으로, 저 SKU는 99% 서비스수준으로" 식으로 **단계별로 차등 선택**하게 바꿉니다. 서비스수준이 높을수록 필요 안전재고도, 비용도 커집니다. 같은 예산이라도 더 정교한 배분이 가능해집니다. 이게 우리 프로젝트의 **진짜 독창적인 부분**입니다(경쟁사가 SME 수준에서 이렇게까지 세분화한다는 근거가 없음, + 강의교재 이론을 정확히 응용했다는 학술적 근거도 있음).

**계산 공식(교재 기준, 이미 확보된 이론)**:
- 서비스수준별 안전계수(Z값): 90%→1.282, 95%→1.645, 99%→2.326 (표준정규분포 분위수함수)
- 필요 안전재고 = Z × σL (σL은 기존 모듈에서 이미 계산하고 있는 리드타임 변동 반영 표준편차)
- 비용 = 안전재고량 × SKU단가 × 재고유지비용률(자본비용8-15%+보관2-5%+진부화1-10%, 합계 15~30% 중 적정값 선택)

**입력**: 기존 ABC-XYZ+σL 모듈의 출력 그대로 사용
**출력**: SKU별로 [선택된 서비스수준, 그 수준의 안전재고량, 비용], 예산 소진 현황, 다음 기능과 연결되는 "선택 안 된 SKU 목록"

**[참고용 AI프롬프트 예시 — 그대로 안 써도 됨]**
```
기존 [언어/프레임워크] 코드에 있는 예산제약 knapsack 함수를 수정해줘.

현재: SKU를 0/1(보호함/포기함)으로만 선택하는 단순 knapsack
변경: 각 SKU마다 서비스수준 옵션 3개(90%/95%/99%)를 만들고,
- 안전재고량 = Z값(90%=1.282, 95%=1.645, 99%=2.326) × σL(기존 계산값)
- 비용 = 안전재고량 × 단가 × 재고유지비용률(0.2 정도로 일단 고정)
이 정보를 바탕으로, SKU마다 [선택안함, 90%, 95%, 99%] 중 하나를 고르는
다중선택 배낭문제(multi-choice knapsack)로 바꿔줘.
목적함수는 "선택된 서비스수준들의 보호가치 합 최대화", 제약은 "총비용 ≤ 예산".
결과는 SKU별 선택된 옵션, 전체 예산소진율, 선택 안 된 SKU 목록을 출력해줘.
```

### 기능 2 (보조 검증기능, 우선순위 중) — GSCPI 자동조회 + 위기배수 추천

**왜 필요한가**: 지금 위기배수(1.0/1.18/1.8/2.2배)는 고정값입니다. 이걸 외부 공급망압력지수(GSCPI, 미국 뉴욕연준이 매월 무료로 공개)와 비교해서 "지금이 어느 위기단계에 가까운지" 참고치를 보여주는 기능입니다.

**⚠️주의**: 이건 **핵심차별점이 아니라 보조기능**입니다(검토 결과 "지표 모니터링 자체는 차별점이 아니다"로 판단됨). 시간이 부족하면 이 기능은 건너뛰어도 전체 로직에 지장 없습니다. 또한 **위기배수 1.8배/2.2배는 "정확히 며칠에서 며칠로 증가했는지"의 정밀 출처가 끝내 확인되지 않았고, 업계 추정 수준으로 최종 정리됐습니다**(다른 확실한 데이터인 GSCPI 사상최고치·운임7배폭등으로 "극심한 위기였다"는 정황은 뒷받침됨). 배수값 자체는 그대로 쓰되, 하드코딩하지 말고 설정값으로 분리해주세요 — 발표/보고서에서 "정확히 몇일에서 몇일로 늘었다"는 식의 정밀 주장은 하지 않기로 했기 때문에, 코드 내 변수명이나 주석에도 "정밀검증된 값"이 아니라 "업계추정 기반 캘리브레이션 값"이라고 표기해주시면 좋습니다.

**데이터 소스**: https://www.newyorkfed.org/medialibrary/research/interactives/gscpi/downloads/gscpi_data.xlsx (무료, API키 불필요, 월 1회 갱신 — "실시간"이 아님에 유의)

**[참고용 AI프롬프트 예시]**
```
GSCPI 데이터(https://www.newyorkfed.org/medialibrary/research/interactives/gscpi/downloads/gscpi_data.xlsx)를
다운로드해서 최신 월별 수치를 가져오는 함수를 만들어줘.

가져온 값을 아래 잠정 기준표와 비교해서 위기단계를 "참고 추천"으로 보여줘
(자동 적용 아님, 사용자가 최종 확인 후 선택):
- GSCPI < 0.5: Normal(1.0배)
- 0.5~2.0: Warning(1.18배)
- 2.0~4.0: Crisis(1.8배)
- 4.0 이상: Extreme(2.2배)

이 기준값은 설정파일로 분리해줘(나중에 숫자가 바뀔 수 있음).
```

### 기능 3 (저비용, 우선순위 낮음) — 포기 SKU + 예상손실액 명시

**왜 필요한가**: 기능1(다중선택knapsack)을 만들면 자동으로 "예산 안에 못 들어간 SKU"가 생깁니다. 이걸 숨기지 않고 화면에 별도로 보여주고, 그 SKU들을 보호 안 했을 때 예상 손실액(결품비용 등)도 같이 보여주는 UI/로직입니다. 구현 비용이 낮아서 시간 남으면 추가하는 식으로 진행하면 됩니다.

**[참고용 AI프롬프트 예시]**
```
기능1(다중선택knapsack)의 출력 중 "선택 안 된 SKU 목록"을 받아서,
각 SKU의 예상손실액(결품 시 손실 = 평균수요 × 단가 × 결품기간 같은 간단한 근사식)을
계산해 합계와 함께 별도 섹션으로 보여주는 화면/리포트를 만들어줘.
```

## 3. 우선순위 정리

1. **기능1(서비스수준 차등 knapsack)** — 이것부터, 가장 중요
2. **기능3(포기SKU+손실액)** — 기능1 끝나면 바로 추가 가능, 비용낮음
3. **기능2(GSCPI연동)** — 시간 남으면, 없어도 전체 로직엔 지장 없음

## 4. 진행 시 주의사항

- 기존 4모듈(ABC-XYZ, σL안전재고, 위기배수, 예산knapsack)의 **인터페이스(입출력 형식)는 가능한 유지**해주세요. 기능1은 knapsack 부분만 바꾸는 거지 전체를 새로 짜는 게 아닙니다.
- Before/After 시뮬레이션 수치(기존 69.1% 개선)는 **기능1 적용 후 다시 계산해야 합니다**. 구버전 수치를 보고서에 그대로 쓰면 안 됩니다.
- 위기배수 근거수치(108일/110일)는 **정밀출처 없음으로 최종 확정**됐습니다(재검증 끝남, 더 안 파셔도 됩니다). 코드에 그 숫자를 직접 박아넣지 말고, 설정값으로 분리해주시면 나중에 숫자만 바꾸면 됩니다.
