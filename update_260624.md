## 1. 목적

`doc/new`의 최신 문서 3종을 참고하여 현재 SCM 시스템의 기획·설명·개발 문서를 갱신한다.  
다만 최신 문서에 적힌 과거 개발환경과 미완료 상태가 현재 구현보다 뒤처지는 경우에는 **현재 시스템을 기준으로 유지**한다.

이번 작업의 핵심은 기존 기능을 되돌리는 것이 아니라 다음 내용을 정확하게 반영하는 것이다.

- 제품 포지셔닝을 위기 발생 후 대응보다 **연간 위기 대비(Crisis Preparedness)** 중심으로 정리
- 제한된 예산에서 보호 조합을 찾는 **역방향 의사결정 문제**를 핵심 차별점으로 명시
- Knapsack 알고리즘 자체가 선행기술임을 인정하고, 시스템 결합 구조를 차별점으로 설명
- 항로 기반 동시위험과 예산 집중도를 보여주는 **항로 집중도 경고**를 신규 검토 항목으로 추가
- 구현 상태, 실행환경, API 공급자와 안전재고 계산 원칙을 현재 코드에 맞게 정정

| 항목 | 유지 기준 |

| 실행 명령 | `py`가 아닌 `uv run ...` 유지 |

| 포기 SKU | 구현 완료 상태 유지 |

| 위기배수 | 평균 리드타임 `L`에만 적용하고 `σL`에는 중복 적용하지 않는 수정 코드 유지 |
| 배포 | GitHub `heedo-incheninu/scm`과 Streamlit Cloud 운영환경 유지 |

## 4. 반영할 신규 변경사항

### 4.1 제품 포지셔닝

기존의 “위기 발생 시 대응” 표현을 다음 방향으로 보완한다.

> 해운 위기 발생 전에 예산과 SKU 보호 수준을 점검하는 연간 위기 대비 의사결정 도구

앱의 계산 기능은 변경하지 않고 `readme.md`, 발표 설명, 도움말과 AI 조언 문구에서 일관되게 사용한다.

### 4.2 차별점 표현

다음 내용을 핵심 차별점으로 정리한다.

1. 기존 목표달성형 계산은 목표 서비스수준에서 필요한 재고를 구한다.
2. 현재 시스템은 제한된 예산에서 SKU별 보호 수준 조합을 구한다.
3. Knapsack 알고리즘 자체는 IBM 특허 `US6078900`의 선행기술 범위임을 인정한다.
4. 차별점은 해상 리스크, ABC-XYZ, 서비스수준 선택, 예산제약, AI 설명과 SME 경량화의 결합에 있다.

“세계 최초”, “경쟁사에 없는 기능”, “AI가 자동 결정한다”와 같은 검증되지 않은 표현은 사용하지 않는다.

### 4.3 항로 집중도 경고

현재 시스템에 없는 신규 후보 기능으로 별도 관리한다.

- SKU 데이터에 선택형 `route` 컬럼 추가 검토
- 항로별 배정 예산 합계와 비중 계산
- `HHI = Σ(항로별 예산 비중²)` 계산
- HHI 구간별 양호·주의·위험 표시
- 항로별 예산 비중 막대그래프 제공
- 데이터에 항로가 없으면 기능을 비활성화하고 기존 CSV 호환성 유지

초기 경고 기준은 검증 전 임시 설정으로 관리한다.

| HHI | 초기 표시 |
|---:|---|
| 0.25 이하 | 양호 |
| 0.25 초과~0.50 이하 | 주의 |
| 0.50 초과 | 위험 |

이 기능은 안전재고 수량을 자동 증감시키지 않고, 동일 항로에 보호 예산이 집중된 위험을 설명하는 **보조 경고 지표**로만 사용한다.

### 4.4 안전재고 설명 정합성

신규 문서의 단순 표현인 `Z × σL`을 현재 구현 공식으로 정정한다.

```text
SS = z × √(L × σd² + d² × σL²)
```

- 위기배수는 평균 리드타임 `L`에만 적용
- 리드타임 표준편차 `σL`에는 위기배수를 적용하지 않음
- 재주문점은 증가한 평균 리드타임과 안전재고를 함께 사용
- 서비스수준 90%·95%·99% 후보도 동일한 수정 산식을 재사용

## 7. 테스트 기준

- 위기배수는 평균 리드타임에만 적용되어야 한다.
- 서비스수준 다중선택 배분 결과가 예산을 초과하지 않아야 한다.
- SKU당 하나의 서비스수준만 선택되어야 한다.
- 항로 정보가 없는 기존 데이터도 오류 없이 분석되어야 한다.
- 단일 항로에 예산이 모두 배정되면 HHI는 1이어야 한다.
- 예산이 동일 비중으로 여러 항로에 분산되면 HHI가 낮아져야 한다.
- 항로 집중도는 안전재고와 배정 금액을 임의로 변경하지 않아야 한다.
- OpenAI API를 사용할 수 없어도 데모 모드가 정상 동작해야 한다.

## 8. 완료 기준

- 현재 환경과 완료 기능이 신규 문서 때문에 과거 상태로 되돌아가지 않는다.
- README·체크리스트·고도화 문서의 구현 상태가 실제 코드와 일치한다.
- 연간 위기 대비와 역방향 예산 배분이라는 설명이 모든 주요 문서에서 일관된다.
- IBM 선행기술을 인정하면서 시스템 결합 구조의 차별점을 과장 없이 설명한다.
- 항로 집중도 기능은 별도 모듈과 선택형 데이터 구조로 구현할 수 있는 수준까지 설계된다.
- 구현을 진행할 경우 기존 44개 테스트와 신규 항로 위험 테스트가 모두 통과한다.

## 9. 이번 계획에서 제외하는 변경

- OpenAI를 Claude로 되돌리는 작업
- `uv` 실행환경을 `py` 명령으로 교체하는 작업
- 기본 포트를 8503으로 변경하는 작업
- 서비스수준 다중선택 Knapsack을 단순 Fractional Knapsack으로 축소하는 작업
- 포기 SKU와 예상손실 기능 제거 또는 미완료 처리
- 위기배수를 `σL`에 다시 곱하는 계산
- GSCPI 추천값에 따른 시나리오 자동 강제 적용
