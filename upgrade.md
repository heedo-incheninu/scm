# 현재 시스템 고도화 적용안

## 0. 작성 기준

이 문서는 `doc/11_개발자_인수인계서.md`를 중심 기준으로 삼고, `doc` 폴더의 기획·검증·평가기준·근거자료 문서를 보조 자료로 활용하여 현재 SCM Streamlit 시스템에 적용 가능한 고도화 항목을 정리한 계획서이다.

현재 시스템은 이미 ABC-XYZ 분류, 안전재고 계산, 예산 배분 비교, CSV 업로드, Inventory Dashboard형 UI, OpenAI 기반 진단 보조, What-if 시뮬레이션을 포함한다. 따라서 신규 개발은 기존 기능을 대체하기보다, 해커톤 평가에서 차별성이 명확하게 드러나도록 의사결정 알고리즘과 설명 가능성을 강화하는 방향으로 진행한다.

## 1. 현재 시스템에 이미 반영된 기반 기능

- SKU별 중요도 및 변동성 기반 ABC-XYZ 분류
- σL, 즉 리드타임 표준편차를 포함한 안전재고 계산
- 균등·비례·우선순위 예산 배분 비교
- 예산 조건별 보호 수준 시뮬레이션
- CSV 업로드 및 합성 예제 데이터
- 비전문가용 설명 문구, 산식, 데이터 한계 표시
- OpenAI API 기반 진단 보조와 API 미설정 시 데모 모드
- Inventory Dashboard 스타일 UI와 시각화

이 기반 위에서 가장 중요한 고도화 방향은 “단순히 어떤 SKU를 살릴 것인가”에서 “각 SKU를 어느 서비스수준까지 보호할 것인가”로 의사결정 단위를 세분화하는 것이다.

### 1.1 σL 포함 안전재고 구현 여부 확인

현재 시스템에는 σL 포함 안전재고가 이미 구현되어 있다. `src/scm/safety_stock.py`의 `calculate_safety_stock()` 함수는 `lead_time_std_days`를 `lead_time_std_months`로 변환한 뒤, 아래 공식을 사용한다.

```text
SS = z × √(L × σd² + d² × σL²)
```

코드 기준 계산 구조는 다음과 같다.

```text
variance = L × demand_std² + demand_mean² × lead_time_std_months²
safety_stock = z_score × √variance
```

따라서 “σL 포함 안전재고를 새로 추가하는 것”은 현재 고도화 대상이 아니다. 다만 서비스수준 차등 선택형 Knapsack을 구현할 때, 기존 σL 포함 안전재고 공식을 그대로 재사용하여 90%, 95%, 99% 서비스수준별 후보 안전재고를 계산해야 한다.

## 2. 최우선 고도화: 서비스수준 차등 선택형 다중선택 Knapsack

### 적용 필요성

현재 예산 추천은 SKU별 우선순위와 필요 재고를 기준으로 배분한다. 그러나 실제 재고 의사결정에서는 모든 품목을 동일한 보호 수준으로 다루지 않는다. 핵심 품목은 99%, 일반 품목은 95% 또는 90%, 낮은 우선순위 품목은 미선택으로 두는 방식이 더 현실적이다.

따라서 기존 0/1 또는 우선순위 중심 예산 배분을 “SKU별 서비스수준 선택 문제”로 확장한다.

### 제안 모델

SKU마다 다음 선택지를 가진다.

- 미선택
- 90% 서비스수준
- 95% 서비스수준
- 99% 서비스수준

기본 Z값은 다음과 같이 둔다.

- 90%: 1.282
- 95%: 1.645
- 99%: 2.326

서비스수준별 안전재고는 현재 시스템에 구현된 σL 포함 공식을 재사용해 계산한다.

```text
서비스수준별 안전재고 = Z × √(L × σd² + d² × σL²)
서비스수준별 비용 = 안전재고 수량 × 단가 × 재고보유비율
```

여기서 `d`는 월평균 수요, `σd`는 월 수요 표준편차, `L`은 월 단위 평균 리드타임, `σL`은 월 단위 리드타임 표준편차이다. 초기 재고보유비율은 20%를 기본값으로 두되, UI에서 15~30% 범위로 조정 가능하게 만드는 것이 적절하다.

### 구현 방향

신규 모듈을 추가하는 방식을 권장한다.

```text
src/scm/service_level_allocation.py
```

핵심 함수는 다음 역할을 가진다.

- SKU별 서비스수준 후보 생성
- 예산 제약 내 최적 조합 탐색
- SKU당 하나의 선택지만 허용
- 선택된 서비스수준, 안전재고 수량, 투입 비용, 예산 사용률 반환
- 미선택 SKU 목록 반환

알고리즘은 다중선택 Knapsack으로 설계한다. 초기 버전은 SKU 수가 크지 않은 해커톤 MVP 기준이므로 동적계획법 또는 비용 단위 스케일링 방식으로 구현 가능하다.

### UI 반영

`예산 추천` 또는 `전략 비교` 화면에 다음 항목을 추가한다.

- 서비스수준 기반 예산 추천 카드
- 99% / 95% / 90% / 미선택 SKU 수
- SKU별 선택 서비스수준 테이블
- 예산 사용액과 잔여 예산
- 기존 우선순위 배분과의 차이 설명

기존 “예산 절감률” 또는 “보호율” 수치는 새 알고리즘 적용 후 재계산해야 한다. 문서에 언급된 과거 Before/After 수치는 그대로 사용하지 않는다.

### 테스트 기준

- 예산 총액을 초과하지 않아야 한다.
- SKU당 하나의 서비스수준만 선택되어야 한다.
- 예산이 0이면 모든 SKU가 미선택이어야 한다.
- 예산이 충분하면 중요 SKU가 높은 서비스수준으로 선택되어야 한다.
- 동일 예산에서 중요도 가중치가 높은 SKU가 불리해지지 않아야 한다.

## 3. 2순위 고도화: 포기 SKU와 예상손실액 표시

### 적용 필요성

해커톤 심사에서는 “무엇을 추천했는가”만큼 “무엇을 포기했고, 그 결과가 무엇인가”가 중요하다. 제한 예산 상황에서 미선택 SKU를 명시하면 의사결정 도구의 현실성이 크게 높아진다.

### 구현 방향

서비스수준 선택 결과에서 미선택 SKU를 분리하고, 각 SKU의 예상 손실액을 계산한다.

초기 산식은 다음과 같이 단순하고 설명 가능한 방식으로 둔다.

```text
예상손실액 = 평균수요 × 단가 × 예상 결품기간 × 손실계수
```

초기 손실계수는 1.0으로 두고, 추후 마진율·거래처 중요도·대체 가능성 등을 반영할 수 있게 확장한다.

### UI 반영

다음 요소를 추가한다.

- “이번 예산에서 보호하지 못한 SKU” 테이블
- SKU명, 등급, 평균수요, 단가, 예상손실액
- 미선택 사유: 예산 부족, 낮은 중요도, 높은 비용 등
- AI 진단 요약에 “포기 SKU와 예상 손실” 문장 추가

비전문가도 이해할 수 있도록 “이 표는 예산이 부족할 때 감수해야 하는 위험을 보여줍니다”와 같은 설명을 함께 배치한다.

## 4. 3순위 고도화: GSCPI 자동 조회와 위기배수 추천

### 적용 필요성

현재 위기 시나리오는 사용자가 선택하는 방식이다. 여기에 외부 공급망 압력 지표를 연결하면 “왜 이 시나리오를 선택했는가”에 대한 근거가 강화된다.

### 데이터 출처

New York Fed의 GSCPI 데이터를 활용한다.

```text
https://www.newyorkfed.org/medialibrary/research/interactives/gscpi/downloads/gscpi_data.xlsx
```

이 데이터는 월별 지표이며 API 키가 필요하지 않다. 실시간 위기 감지 지표가 아니라 공급망 압력 참고 지표로 표시해야 한다.

### 추천 기준

초기 추천 기준은 다음과 같이 둔다.

| GSCPI 값 | 추천 상태 | 추천 위기배수 |
|---:|---|---:|
| 0.5 미만 | Normal | 1.0 |
| 0.5 이상 2.0 미만 | Warning | 1.18 |
| 2.0 이상 4.0 미만 | Crisis | 1.8 |
| 4.0 이상 | Extreme | 2.2 |

위기배수 1.8, 2.2는 정밀 검증값이 아니라 사례 기반 보정 추정값으로 표시한다. 시스템이 자동 적용하지 않고, 사용자에게 “추천”만 제공해야 한다.

### 구현 방향

- `src/scm/gscpi.py` 신규 모듈 추가
- 데이터 다운로드 및 캐싱
- 최신 월 지표 추출
- 지표 구간별 추천 시나리오 반환
- 네트워크 실패 시 수동 선택 모드 유지

### UI 반영

- 좌측 또는 상단에 “현재 공급망 압력 참고 지표” 카드 추가
- 최신 GSCPI 값과 기준월 표시
- 추천 시나리오와 사용자가 선택한 시나리오를 구분
- “추천값은 참고이며 실제 적용은 사용자가 선택합니다” 문구 표시

## 5. 평가 대응용 설명 강화

`doc/09_평가기준표.md`, `doc/10_근거자료_출처모음.md`, `doc/12_주제_최종결론.md`의 관점을 반영해 다음 설명을 화면과 문서에 추가한다.

### 이 시스템이 증명하는 것

- 제한 예산에서 SKU별 안전재고 보호 우선순위를 계산할 수 있다.
- 위기 상황별 예산 부족 영향을 비교할 수 있다.
- 중요한 품목을 더 높은 서비스수준으로 보호하는 의사결정 구조를 만들 수 있다.
- 비전문가도 산식과 데이터 한계를 확인하며 결과를 해석할 수 있다.

### 이 시스템이 증명하지 않는 것

- 실제 기업의 모든 공급망 리스크를 완전 예측하지 않는다.
- 위기배수가 절대적으로 정확하다고 주장하지 않는다.
- 외부 지표만으로 자동 의사결정을 내리지 않는다.
- 수요예측, Fisher Map, ERP 연동은 현재 MVP 범위가 아니다.

### 차별화 문장

권장 포지셔닝은 다음과 같다.

```text
검증된 재고관리 이론과 공급망 압력 지표를 결합하여,
한국 수입 중소기업이 제한 예산 안에서 SKU별 보호 수준을 결정하도록 돕는
경량 의사결정 보조 도구
```

기존 솔루션도 ABC-XYZ, 안전재고, 대시보드를 제공할 수 있으므로, 차별점은 “한국 수입 SME 대상”, “위기 시나리오”, “서비스수준 차등 예산 배분”, “포기 SKU와 손실 표시”에 둔다.

## 6. 권장 개발 순서

### Phase 1. 서비스수준 후보 테이블 생성

- SKU별 90%, 95%, 99% 안전재고와 비용 계산
- 기존 `src/scm/safety_stock.py`의 σL 포함 안전재고 공식을 재사용
- 기존 안전재고 계산 모듈과 중복되지 않게 연결
- 재고보유비율 파라미터 추가

### Phase 2. 다중선택 Knapsack 구현

- SKU별 하나의 서비스수준만 선택
- 예산 제약 적용
- 결과 테이블 생성
- 기존 우선순위 배분과 비교 가능하게 데이터 구조 정리

### Phase 3. 화면 적용 및 기존 수치 재계산

- 예산 추천 화면에 서비스수준 추천 추가
- 한눈에 보기 카드에 서비스수준별 SKU 수 추가
- 기존 Before/After 수치 폐기 후 새 알고리즘 기준으로 재계산

### Phase 4. 포기 SKU 및 예상손실액 추가

- 미선택 SKU 목록 생성
- 예상손실액 계산
- AI 진단 요약에 반영
- 비전문가용 설명 문구 추가

### Phase 5. GSCPI 추천 기능 추가

- GSCPI 다운로드 및 캐싱
- 최신 지표 기준 추천 시나리오 표시
- 자동 적용이 아닌 수동 확인 방식 유지

### Phase 6. 테스트와 발표용 정리

- 알고리즘 단위 테스트
- 극단 예산 테스트
- 네트워크 실패 테스트
- 해커톤 심사용 설명 섹션 정리
- `checklist.md` 진행 상태 업데이트

## 8. 현재 MVP 범위에서 제외할 항목

다음 기능은 문서상 보류 또는 제외 항목으로 보는 것이 적절하다.

- 정교한 수요예측 모델
- Fisher Map
- ERP 실시간 연동
- 실제 기업 데이터 자동 수집
- 위기배수의 정밀 통계 검증
- 완전 자동 발주 의사결정

이 항목을 추가하려면 `readme.md`, `checklist.md`, 기획 문서를 함께 갱신해야 한다.

## 9. 완료 기준

고도화가 완료되었다고 판단하려면 다음 조건을 만족해야 한다.

- 예산 추천에서 SKU별 90%, 95%, 99%, 미선택 결과가 표시된다.
- 예산 총액을 초과하지 않는 추천 결과가 나온다.
- 미선택 SKU와 예상손실액이 화면에 직접 표시된다.
- GSCPI 최신 값 또는 실패 시 fallback 상태가 명확히 표시된다.
- 산식, 데이터 출처, 한계가 화면에서 확인 가능하다.
- 기존 균등·비례·우선순위 배분과 신규 서비스수준 배분을 비교할 수 있다.
- 관련 테스트가 추가되고 통과한다.
- `checklist.md`에 모듈별 진행 상태가 업데이트된다.
