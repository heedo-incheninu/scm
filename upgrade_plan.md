# 사용자 이해 중심 고도화 개발 계획서

## 0. 목표

`upgrade.md`의 고도화 항목을 실제 개발로 옮길 때, 가장 중요한 목표는 사용자가 전체 의사결정 흐름을 쉽게 이해하도록 만드는 것이다.

즉, 단순히 “정확한 계산 결과”를 보여주는 것이 아니라 아래 질문에 사용자가 순서대로 답을 얻도록 설계한다.

1. 지금 어떤 데이터를 보고 있는가?
2. 현재 공급망 위험은 어느 정도인가?
3. 어떤 SKU가 중요한가?
4. 각 SKU는 얼마만큼의 안전재고가 필요한가?
5. 예산이 부족하면 어떤 품목을 어느 수준까지 보호해야 하는가?
6. 보호하지 못한 SKU의 위험과 예상 손실은 무엇인가?
7. 그래서 지금 무엇을 결정해야 하는가?
8. 보호 예산이 한 항로에 과도하게 집중되어 있지는 않은가?

### 2026-06-24 추가 구현 원칙

- 제품의 `route` 항로 정보는 선택 입력으로 받는다.
- 항로 정보가 없어도 기존 분석 흐름을 중단하지 않는다.
- 항로 HHI는 동시위험을 설명하는 보조 지표이며 안전재고·배정액을 변경하지 않는다.
- 현재 OpenAI Responses API, `uv`, 포트 8501, 서비스수준 다중선택 Knapsack을 유지한다.

## 1. 개발 원칙

### 1.1 흐름 우선

기능을 화면에 나열하지 않고, 사용자가 따라갈 수 있는 단계형 흐름으로 구성한다.

```text
데이터 확인 → 위험 시나리오 선택 → 품목 중요도 확인 → 안전재고 계산 → 예산 배분 → 포기 SKU 확인 → 의사결정 요약
```

각 화면은 “이 단계에서 무엇을 판단해야 하는지”를 먼저 설명하고, 그 다음 표·그래프·계산 결과를 보여준다.

### 1.2 산식과 결과를 분리하지 않기

사용자가 숫자를 볼 때 바로 근거를 이해할 수 있도록, 주요 계산 결과 옆에 짧은 산식 설명을 배치한다.

예를 들어 안전재고 화면에는 다음 설명을 함께 둔다.

```text
안전재고 = z × √(L × σd² + d² × σL²)
```

현재 시스템에는 σL, 즉 리드타임 표준편차를 포함한 안전재고 계산이 이미 구현되어 있다. 고도화에서는 이 공식을 새로 만드는 것이 아니라, 서비스수준 90%, 95%, 99% 후보 계산에 재사용한다.

### 1.3 추천과 한계를 함께 표시

추천 결과만 보여주면 사용자가 과신할 수 있다. 따라서 모든 주요 추천 화면에는 다음 세 가지를 함께 보여준다.

- 추천 결과
- 추천 근거
- 데이터 한계

특히 GSCPI 기반 위기배수는 자동 의사결정이 아니라 참고 지표로 표시한다.

## 2. 권장 사용자 여정

### Step 1. 한눈에 보기

사용자가 처음 보는 화면이다. 전체 상황을 30초 안에 이해하게 만드는 것이 목적이다.

표시 항목:

- 전체 SKU 수
- 추가 안전재고 필요 SKU 수
- 필요 예산 총액
- 현재 선택 예산
- 우선 보호 SKU 수
- 미보호 SKU 수
- 예상 손실 총액
- 선택된 공급망 시나리오

화면 상단에는 짧은 안내 문구를 둔다.

```text
이 대시보드는 제한된 예산 안에서 어떤 SKU를 어느 수준까지 보호해야 하는지 보여줍니다.
```

### Step 2. 데이터 확인

CSV 업로드 또는 예제 데이터를 사용자가 명확히 인식하게 한다.

표시 항목:

- 현재 사용 데이터 출처
- 예제 데이터 여부
- SKU 수
- 기간
- 주요 컬럼 설명
- 데이터 한계

필수 설명:

```text
현재 데이터는 의사결정 구조를 검증하기 위한 합성 또는 업로드 데이터입니다.
실제 발주 전에는 기업 내부 수요, 단가, 리드타임 데이터를 확인해야 합니다.
```

### Step 3. 공급망 위험 시나리오

사용자가 “왜 이 위기 조건을 선택했는지” 이해하도록 한다.

구성:

- Normal / Warning / Crisis / Extreme 시나리오 카드
- 각 시나리오의 리드타임 배수
- GSCPI 최신값이 있으면 추천 시나리오 표시
- 사용자가 최종 시나리오를 직접 선택

중요 원칙:

- GSCPI는 추천만 한다.
- 선택은 사용자가 한다.
- 위기배수는 정밀 예측값이 아니라 시나리오 가정으로 설명한다.

### Step 4. 품목 중요도 이해

ABC-XYZ 등급을 비전문가도 이해할 수 있게 풀어쓴다.

표시 항목:

- ABC: 매출 또는 수요 규모 기준 중요도
- XYZ: 수요 변동성 기준 안정성
- 9개 등급별 SKU 목록
- 등급별 관리 의미

예시 설명:

```text
AX 품목은 중요도는 높고 수요는 안정적이므로 우선 보호 대상입니다.
CZ 품목은 중요도는 낮고 수요 변동성이 커서 제한 예산에서는 후순위가 될 수 있습니다.
```

### Step 5. 안전재고 필요량 이해

현재 시스템의 σL 포함 안전재고 계산을 사용자가 이해하도록 설명한다.

표시 항목:

- SKU별 권장 안전재고
- 현재 재고
- 부족 수량
- 필요 예산
- 리드타임 평균
- 리드타임 표준편차
- 수요 표준편차

핵심 설명:

```text
수요가 불안정하거나 배송 기간의 변동성이 커질수록 더 많은 안전재고가 필요합니다.
```

### Step 6. 서비스수준 기반 예산 배분

`upgrade.md`의 최우선 고도화 항목이다. 기존 “살릴 SKU 선택”에서 “SKU별 보호 수준 선택”으로 바꾼다.

서비스수준 선택지:

- 미선택
- 90%
- 95%
- 99%

표시 항목:

- SKU명
- ABC-XYZ 등급
- 추천 서비스수준
- 추천 안전재고
- 투입 비용
- 예산 사용률
- 선택 사유

사용자 설명:

```text
예산이 부족할 때 모든 품목을 100% 보호할 수는 없습니다.
이 화면은 중요한 품목은 높은 서비스수준으로, 덜 중요한 품목은 낮은 수준 또는 미선택으로 배분합니다.
```

### Step 7. 포기 SKU와 예상손실액

제한 예산 의사결정의 현실성을 보여주는 화면이다.

표시 항목:

- 미선택 SKU 목록
- 미선택 사유
- 평균수요
- 단가
- 예상 결품기간
- 예상손실액

초기 산식:

```text
예상손실액 = 평균수요 × 단가 × 예상 결품기간 × 손실계수
```

주의 문구:

```text
예상손실액은 의사결정 비교를 위한 추정값이며 실제 손실액을 보장하지 않습니다.
```

### Step 8. 항로 집중 위험 확인

서비스수준 보호 예산을 항로별로 집계하고 `HHI = Σ(항로별 예산 비중²)`를 계산한다.
위험 시나리오와 전략 비교 화면에서 항로별 예산 비중, HHI와 경고 등급을 표시한다.

- HHI 0.25 이하: 양호
- HHI 0.25 초과~0.50 이하: 주의
- HHI 0.50 초과: 위험

항로 정보가 일부만 있으면 정보 확인율을 함께 표시하고, 전혀 없으면 계산하지 않는다.
경고 문구에는 공급사·항만·대체 항로와 함께 판단해야 한다는 한계를 명시한다.

### Step 9. 전략 비교

사용자가 신규 방식의 의미를 기존 방식과 비교하게 한다.

비교 대상:

- 균등 배분
- 비례 배분
- 기존 우선순위 배분
- 신규 서비스수준 차등 배분

비교 지표:

- 보호 SKU 수
- 중요 SKU 보호율
- 예산 사용률
- 미보호 SKU 수
- 예상손실액
- 평균 서비스수준

핵심 질문:

```text
같은 예산이라면 어떤 전략이 중요한 SKU를 더 잘 보호하는가?
```

### Step 10. AI 의사결정 요약

AI는 계산을 대체하는 것이 아니라 사용자가 결과를 해석하도록 돕는 역할로 둔다.

요약에 포함할 내용:

- 현재 시나리오
- 예산 조건
- 가장 중요한 보호 대상
- 미보호 SKU와 예상 손실
- 추천 액션
- 데이터 한계

권장 문장 구조:

```text
현재 Crisis 시나리오와 예산 40% 조건에서는 AX 등급 SKU를 우선 보호하는 것이 적절합니다.
다만 CZ 등급 일부 SKU는 미보호 상태이며, 예상손실액은 약 ₩X입니다.
```

## 3. 화면 구조 개편안

권장 메뉴 구조는 다음과 같다.

```text
1. 한눈에 보기
2. 데이터 확인
3. 위험 시나리오
4. 품목 중요도
5. 안전재고
6. 예산 추천
7. 포기 SKU
8. 전략 비교
9. AI 요약
```

각 메뉴는 독립적인 기능 화면이 아니라 하나의 의사결정 흐름으로 연결되어야 한다. 메뉴 상단에는 “현재 단계 / 전체 단계”를 표시한다.

예:

```text
4 / 9단계: 품목 중요도 확인
```

## 4. 개발 단계

### Phase 1. 현재 흐름 정리

목표:

- 기존 화면을 사용자 여정 기준으로 재정렬한다.
- 각 화면 상단에 “이 단계에서 판단할 것” 설명을 추가한다.

작업:

- `app.py`의 페이지 구성 점검
- 화면별 안내 문구 추가
- 현재 계산 결과와 산식 설명 위치 정리

완료 기준:

- 사용자가 메뉴 순서대로 보면 전체 흐름을 이해할 수 있다.
- 각 화면에 최소 1개의 목적 설명이 있다.

### Phase 2. 서비스수준 배분 설계

목표:

- 90%, 95%, 99%, 미선택 옵션을 생성한다.
- 기존 σL 포함 안전재고 공식을 재사용한다.

작업:

- `src/scm/service_level_allocation.py` 신규 작성
- SKU별 서비스수준 후보 테이블 생성
- 예산 제약 내 최적 조합 선택
- 결과 컬럼 표준화

완료 기준:

- SKU당 하나의 서비스수준만 선택된다.
- 예산 총액을 초과하지 않는다.
- 결과에 추천 사유가 포함된다.

### Phase 3. 예산 추천 화면 개편

목표:

- 사용자가 예산 조건에 따라 보호 수준이 어떻게 달라지는지 이해한다.

작업:

- 서비스수준별 SKU 수 카드 추가
- 추천 결과 테이블 추가
- 기존 우선순위 배분과 신규 배분 비교
- 예산 사용률과 잔여 예산 표시

완료 기준:

- 같은 화면에서 “무엇을 얼마나 보호하는지” 확인 가능하다.
- 기존 방식과 신규 방식 차이가 명확히 보인다.

### Phase 4. 포기 SKU와 예상손실액 추가

목표:

- 예산 부족의 결과를 사용자가 직접 확인한다.

작업:

- 미선택 SKU 추출
- 예상손실액 계산
- 포기 사유 생성
- AI 요약에 반영

완료 기준:

- 미보호 SKU 목록이 표시된다.
- 예상손실액 총액과 SKU별 손실액이 표시된다.
- 손실액 한계 문구가 함께 표시된다.

### Phase 5. GSCPI 추천 기능 추가

목표:

- 위기 시나리오 선택에 외부 근거를 제공한다.

작업:

- `src/scm/gscpi.py` 신규 작성
- GSCPI 데이터 다운로드 및 캐싱
- 최신 지표 기준 추천 시나리오 반환
- 네트워크 실패 fallback 처리

완료 기준:

- 최신 GSCPI 값 또는 실패 상태가 표시된다.
- 추천 시나리오와 사용자 선택 시나리오가 구분된다.
- 자동 적용이 아닌 참고 지표로 설명된다.

### Phase 6. 전략 비교 고도화

목표:

- 신규 서비스수준 배분이 기존 방식보다 어떤 점에서 나은지 비교한다.

작업:

- 비교 대상에 서비스수준 배분 추가
- 예상손실액 비교 추가
- 중요 SKU 보호율 비교 추가
- 예산별 변화 그래프 추가

완료 기준:

- 균등·비례·우선순위·서비스수준 배분을 같은 기준으로 비교할 수 있다.

### Phase 7. 테스트 및 문서 업데이트

목표:

- 기능 안정성과 문서 일관성을 확보한다.

작업:

- 서비스수준 배분 테스트 추가
- 극단 예산 테스트 추가
- GSCPI fallback 테스트 추가
- `checklist.md` 업데이트
- `readme.md` 개발 순서 반영

완료 기준:

- `uv run pytest -q` 통과
- `uv run ruff check .` 통과
- `checklist.md`에 진행 상태가 반영된다.

## 5. 사용자 이해를 위한 문구 배치 기준

모든 주요 화면은 다음 구조를 따른다.

```text
1. 이 화면의 목적
2. 사용자가 봐야 할 핵심 숫자
3. 계산 또는 추천 근거
4. 결과 해석
5. 데이터 한계
```

예산 추천 화면 예시:

```text
이 화면은 제한된 예산 안에서 SKU별 보호 수준을 추천합니다.
99%는 매우 높은 보호 수준이고, 90%는 상대적으로 낮은 보호 수준입니다.
예산이 부족하면 모든 SKU를 99%로 보호할 수 없기 때문에 중요 품목에 예산을 먼저 배정합니다.
```

## 7. 최종 완료 기준

고도화 개발이 완료되었다고 판단하려면 다음 조건을 만족해야 한다.

- 사용자가 메뉴 순서만 따라가도 전체 의사결정 흐름을 이해할 수 있다.
- σL 포함 안전재고 공식이 화면에 설명되고 계산에 사용된다.
- 서비스수준 90%, 95%, 99%, 미선택 추천 결과가 표시된다.
- 예산 부족으로 보호하지 못한 SKU와 예상손실액이 표시된다.
- GSCPI는 참고 지표로 표시되고 사용자가 최종 시나리오를 선택한다.
- 기존 배분 방식과 신규 서비스수준 배분을 비교할 수 있다.
- 모든 주요 화면에 산식, 출처, 데이터 한계가 배치된다.
- 테스트와 린트가 통과한다.
- `checklist.md`에 모듈별 진행 상황이 업데이트된다.
