# 해상물류 SCM 리스크 진단 플랫폼

해상 물류 리드타임의 변동성을 반영하여 위기 상황에서 SKU별 안전재고와 예산 배분 우선순위를 제안하는 SCM 의사결정 지원 플랫폼입니다. 주요 대상은 전용 SCM 시스템 도입이 어려운 국내 수입 중소기업이며, Streamlit 기반의 경량 PoC를 목표로 합니다.

이 문서는 `Plan.md`를 바탕으로 구현한 PoC의 개발·실행 기준서입니다. 실제 검증 상태는 `checklist.md`에서 관리합니다.

## 1. 해결하려는 문제

해상 리드타임 변동성 자체는 기업이 제거하기 어려운 외부 조건입니다. 이 프로젝트는 위기 발생 시 모든 SKU에 동일한 재고 정책을 적용하거나 담당자의 경험에만 의존하는 내부 의사결정 문제를 해결합니다.

핵심 흐름은 다음과 같습니다.

1. SKU를 매출 기여도와 수요 변동성으로 분류합니다.
2. 리드타임 표준편차(σL)를 반영하여 시나리오별 안전재고를 계산합니다.
3. 제한된 예산을 SKU 중요도에 따라 우선 배분합니다.
4. SKU별 90%·95%·99%·미보호 서비스수준 중 하나를 선택합니다.
5. 보호하지 못한 SKU와 예상손실액을 확인합니다.
6. 계산 결과를 AI가 실무자가 이해할 수 있는 진단문으로 설명합니다.
7. 기존 방식과 비교하여 결품 위험 개선 효과를 검증합니다.

## 2. 대상 환경과 데이터

- Python 3.11 이상
- Streamlit
- SQLite
- OpenAI Responses API 및 API를 사용할 수 없을 때의 데모 모드
- GSCPI XLSX 조회를 위한 openpyxl
- 30개 SKU, SKU별 24개월 판매 이력
- 9개 ABC-XYZ 셀에 분산된 SCM 이론 기반 시뮬레이션 데이터

SKU 단위 기업 재고 데이터는 영업비밀로 공개되지 않는 경우가 많으므로, 이 PoC는 수요 패턴을 시뮬레이션으로 구성합니다. 리드타임과 위기 배수는 공개된 실측 근거를 사용하며, 실제 기업 데이터로 검증하지 못한 한계를 결과와 보고서에 명시해야 합니다.

### 프로젝트 구조

```text
app.py                 Streamlit 진입점
src/scm/               분류·안전재고·배분·진단 계산 모듈
scripts/init_db.py     SQLite 표본 데이터 초기화
scripts/generate_csv_samples.py  업로드용 가상 CSV 5종 생성
tests/                 단위·통합·UI 스모크 테스트
data/scm.db            실행 시 생성되는 로컬 데이터베이스
csv/                   업로드 형식의 가상 기업 데이터 5종
.streamlit/config.toml UI 테마 설정
```

## 3. 개발 모듈과 구현 순서

모듈은 아래 순서대로 개발합니다. 뒤쪽 모듈은 앞쪽 모듈의 출력과 검증 결과를 입력으로 사용합니다.

### 모듈 1. 프로젝트 기반 및 데이터베이스

- 목적: Streamlit 애플리케이션과 분석 모듈이 공유할 실행 환경 및 SQLite 구조를 준비합니다.
- 입력: SKU, 판매 이력, 단가, 재고와 리드타임 관련 기초 데이터 정의
- 출력: 초기화된 데이터베이스와 모듈에서 재사용할 데이터 접근 계층
- 완료 조건: 빈 환경에서 데이터베이스를 초기화하고 동일한 스키마를 재현할 수 있어야 합니다.

### 모듈 2. SKU 시뮬레이션 데이터

- 목적: 실제 기업 데이터의 비공개 한계를 보완하는 재현 가능한 PoC 데이터를 생성합니다.
- 입력: 30개 SKU와 24개월 수요 패턴 생성 규칙
- 출력: ABC-XYZ의 9개 셀을 모두 포함하는 판매 이력 데이터
- 완료 조건: 같은 시드에서 같은 데이터가 생성되고, 누락값·음수·중복 키 검사를 통과해야 합니다.
- CSV 업로드: 제품 정보와 월별 판매 이력을 한 파일로 불러오며 형식 오류를 화면에서 설명합니다.
- 예제 데이터: 전자부품·식품원료·자동차부품·의료소모품·패션상품 5개 CSV를 제공합니다.

### 모듈 3. ABC-XYZ 분류

- 목적: 매출 기여도와 수요 변동성을 결합하여 SKU의 관리 중요도를 정합니다.
- 입력: SKU별 판매량, 단가, 24개월 판매 이력
- 출력: 각 SKU의 ABC 등급, XYZ 등급, 결합 셀과 우선순위 가중치
- 완료 조건: 모든 SKU가 정확히 하나의 셀에 배정되고, 분류 결과와 실제 예산 계산의 우선순위가 일치해야 합니다.

### 모듈 4. 안전재고 및 위기 시나리오

- 목적: 리드타임 표준편차(σL)를 명시적으로 반영한 시나리오별 안전재고를 계산합니다.
- 입력: SKU별 수요 정보, 기준 리드타임, 리드타임 변동성, 서비스 수준과 시나리오 배수
- 출력: 평상시 및 위기 시나리오별 권장 안전재고와 필요 금액
- 완료 조건: 다음 네 시나리오가 같은 계산 규칙으로 재현되고, 시나리오 강도가 높아질수록 필요한 재고가 감소하지 않아야 합니다.

| 시나리오 | 리드타임 배수 | 근거 |
|---|---:|---|
| 평상시 | 1.0배 | 기준값 |
| 중간 충격 | 1.18배 | 2024년 홍해 위기 |
| 고충격 | 1.8배 | 2021~2022년 아시아-유럽 항로 |
| 극단 충격 | 2.2배 | 2021~2022년 아시아-미서부 항로 |

### 모듈 5. 예산 제약 기반 우선순위 배분

- 목적: 전체 권장 재고를 확보할 수 없을 때 제한된 예산을 핵심 SKU부터 배분합니다.
- 입력: 시나리오별 안전재고 필요 금액과 ABC-XYZ 가중치, 사용 가능 예산
- 출력: SKU별 배정 예산, 확보 수량, 충족률, 잔여 예산과 우선순위 근거
- 완료 조건: 총배정액이 예산을 초과하지 않고, 같은 조건에서는 높은 우선순위 SKU가 불리해지지 않아야 합니다.
- 주의사항: 화면에 표시하는 관리 우선순위와 실제 계산 가중치는 반드시 같은 규칙을 사용해야 합니다.
- 비교 기능: 10%·20%·30%·50%·75%·100% 예산에서 균등·비례·우선순위 배분을 비교합니다.

### 모듈 6. OpenAI 진단

- 목적: 분석 결과와 배분 근거를 담당자가 실행할 수 있는 자연어 진단으로 변환합니다.
- 입력: 시나리오, 핵심 SKU, 안전재고 부족량, 배정 결과와 주요 위험 지표
- 출력: 위험 요약, 우선 조치, SKU별 판단 근거와 한계
- 완료 조건: OpenAI API 모드와 데모 모드가 동일한 출력 구조를 사용하고, API 장애가 분석 기능 전체의 실패로 이어지지 않아야 합니다.
- 인증: `OPENAI_API_KEY` 또는 화면의 비밀번호 입력을 사용하며 키를 파일이나 DB에 저장하지 않습니다.
- API: 공식 Python SDK의 Responses API를 사용하고 기본 모델은 `gpt-5.5`입니다.

### 모듈 7. 통합 테스트 및 효과 검증

- 목적: 전체 데이터 흐름을 검증하고 기존 균등 배분 방식과 우선순위 배분 방식을 비교합니다.
- 입력: 동일한 데이터, 시나리오와 예산 조건을 적용한 두 방식의 결과
- 출력: Before/After 비교 지표와 재현 가능한 검증 결과
- 완료 조건: 고정 시드·중간 충격·예산 20% 조건에서 결과를 재현하고, 지표 산식과 비교 기준을 함께 남겨야 합니다.
- 현재 검증값: 위험가중 충족률이 비례 배분 19.84%에서 우선순위 배분 23.75%로 상승해 상대 개선율 19.72%를 기록했습니다. 위험가중치는 `필요 재고금액 × ABC-XYZ 우선순위`입니다.
- `Plan.md`의 69.1%는 산식과 원본 실행 결과가 없어 현재 구현에서 재현 가능한 성과로 사용하지 않습니다.

### 모듈 8. Streamlit UI 및 사용 가이드

- 목적: 분석 과정과 결과를 비전문가도 순서대로 확인할 수 있게 제공합니다.
- 입력: 각 분석 모듈의 표준화된 결과
- 출력: 화이트 모드, 좌측 사이드바 탐색, 대시보드, 상세 분석 화면과 사용 가이드
- 완료 조건: 데이터 입력부터 시나리오 선택, 예산 배분, AI 진단과 비교 결과 확인까지 주요 사용자 흐름이 중단 없이 동작해야 합니다.

### 모듈 9. 서비스수준 배분 및 사용자 이해 고도화

- 목적: 사용자가 “어떤 SKU를 어느 수준까지 보호하고, 무엇을 포기하는지”를 한 흐름으로 이해하게 합니다.
- 입력: σL 포함 안전재고 결과, ABC-XYZ 우선순위, 재고보유비율, 사용 가능 예산
- 출력: SKU별 90%·95%·99%·미보호 추천, 보호 비용, 미보호 SKU, 예상손실액, GSCPI 참고 시나리오
- 완료 조건: 총 보호 비용이 예산을 초과하지 않고, SKU당 하나의 서비스수준만 선택되어야 합니다.
- 비교 기능: 균등·비례·우선순위·서비스수준 배분을 같은 예산 비율에서 비교합니다.
- 화면 흐름: 데이터 확인 → 위험 시나리오 → 품목 중요도 → 필요 재고 → 예산 추천 → 포기 SKU → 전략 비교 → AI 요약 순서로 안내합니다.

## 4. 현재 범위에서 제외한 기능

- 수요예측: 보조 모듈로 검토했으나 현재 버전에서는 구현하지 않습니다.
- Fisher Map: 보조 모듈로 검토했으나 현재 버전에서는 구현하지 않습니다.

두 기능을 다시 포함할 때는 핵심 모듈의 계산 결과를 변경하지 않는 독립 기능으로 설계하고, `checklist.md`에 새 작업 범위와 완료 조건을 먼저 추가합니다.

## 5. 진행 상황 관리

모든 개발 진행 상황은 [`checklist.md`](checklist.md)에서 관리합니다.

- `[ ]`: 시작 전 또는 미완료
- `[~]`: 진행 중
- `[x]`: 구현과 검증을 모두 완료
- `[-]`: 현재 범위에서 제외

작업할 때 다음 규칙을 적용합니다.

1. 작업 시작 전에 해당 항목을 `[~]`로 변경합니다.
2. 구현만 끝난 상태는 완료로 처리하지 않습니다. 관련 테스트와 문서 확인 후 `[x]`로 변경합니다.
3. 요구사항이나 계산 규칙이 바뀌면 이 문서와 `checklist.md`를 같은 변경에서 함께 갱신합니다.
4. 버그는 발견 내용, 수정 내용과 회귀 테스트를 해당 모듈 아래에 기록합니다.
5. 외부 요인으로 보류한 항목은 원인과 재개 조건을 함께 기록합니다.
6. 모듈 완료 여부는 하위 항목이 모두 완료된 경우에만 완료로 표시합니다.

## 6. 설치·실행·테스트 정보

프로젝트 루트에서 다음 명령을 실행합니다. `uv`가 없다면 먼저 [uv 공식 설치 안내](https://docs.astral.sh/uv/getting-started/installation/)에 따라 설치합니다.

```powershell
# Python 환경과 전체 개발 의존성 설치
uv sync --all-groups

# 고정 시드 표본 DB 생성 또는 재생성
uv run python scripts/init_db.py --replace

# 웹 애플리케이션 실행
uv run streamlit run app.py

# 테스트와 정적 검사
uv run pytest -q
uv run ruff check .
```

OpenAI API를 사용할 때만 현재 PowerShell 세션에 키와 모델을 설정합니다. 키가 없거나 호출이 실패하면 앱은 자동으로 데모 진단을 사용합니다. ChatGPT 구독과 OpenAI API 결제는 별도입니다.

```powershell
$env:OPENAI_API_KEY="발급받은-키"
$env:SCM_OPENAI_MODEL="gpt-5.5"
uv run streamlit run app.py
```

GSCPI 자동 조회를 끄고 수동 시나리오 선택만 테스트하려면 다음 환경변수를 사용합니다.

```powershell
$env:SCM_SKIP_GSCPI="1"
```

업로드용 가상 CSV를 다시 만들려면 다음 명령을 실행합니다.

```powershell
uv run python scripts/generate_csv_samples.py
```
